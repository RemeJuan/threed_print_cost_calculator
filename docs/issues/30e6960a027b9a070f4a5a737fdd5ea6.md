## Suspected cause
This ANR occurs when the app experiences a fatal uncaught exception and the Android framework's `KillApplicationHandler` attempts to report the crash to the ActivityManager service. The binder call to `handleApplicationCrash` is synchronous and blocks the main thread. If the system server is slow to respond (e.g., under memory pressure, during a system-level deadlock, or when the app is already in a corrupted state), the main thread hangs past the ANR threshold.

This is a system-level issue during crash handling — the app is already dying when the ANR occurs. The ANR is a secondary symptom of the system server being unresponsive during crash reporting, not the primary cause of the app failure.

## Reproduction
Not locally reproducible. Single occurrence on one device.

## Root cause
This is a known Android OS behavior where crash reporting can trigger ANRs on some devices. The `IActivityManager.handleApplicationCrash` binder transaction blocks the main thread while the system server processes the crash report. On Pixel devices running Android 12, this has been observed to cause ANRs when the system server is under load.

The actual trigger for the original uncaught exception is unknown (the ANR itself masks the original crash). However, given the low volume (2 events, 1 user), this is likely a one-off occurrence.

## Why this is not-fixable
1. **System-level issue**: We cannot modify Android's crash handling mechanism or binder transaction behavior.
2. **Already-dying process**: The app is already experiencing a fatal exception when this ANR occurs — we can't prevent the ANR without preventing the original crash.
3. **No code change available**: There is no application-level code change that can prevent the `KillApplicationHandler` from calling `handleApplicationCrash`.
4. **Known Android limitation**: This is a documented Android framework behavior — synchronous binder calls to system services during crash handling can timeout on some devices.
